## 前言

最近挖到了一个很有意思的开源项目，叫 MiniMind-O，是 Jingyao Gong 做的一个全模态模型。

一开始看到 0.1B 参数的时候我是拒绝的 (´-ω-`)b... 这个体量能干嘛？但仔细一看发现它居然同时支持 **文本、语音、图像、视频** 四种模态的输入和输出，而且号称可以在消费级显卡上跑。本着"试试又不会少块肉"的心态就折腾了一下。

结果嘛...确实有点东西 (｀・ω・´)

---

## 我的设备情况

手头有两台机器：

| 设备 | 配置 |
|------|------|
| 🖥️ 台式机 | CPU 20线程 + 32GB 内存 + RTX 5060 8GB |
| 💻 **轻薄本** | **AMD Radeon 880M 核显** + **32GB 内存** |

本来想着台式机肯定没问题，轻薄本嘛...核显跑大模型？感觉就像是拿自行车上高速 ( ´ ▽ ` )ﾉ

但实际上两台都跑起来了，而且轻薄本的表现让我有点意外。

---

## 0.1B 究竟意味着什么

现在市面上主流的大模型动辄 7B、13B、70B、甚至 671B... 0.1B（也就是 1 亿参数）放在里面基本上就是"蝼蚁"级别。

但 MiniMind-O 的巧妙之处在于它的架构设计：

```
主模型（Thinker + Talker）：113M 参数
├── Thinker（语言核心）：63.91M
├── Talker（语音生成）：47.05M
├── 音频投影层：0.99M
└── 视觉投影层：1.18M

外挂编码器（冻结，不参与训练）：
├── SenseVoice 语音编码器：234M
├── SigLIP2 视觉编码器：95M
└── Mimi 音频编解码器：96M
```

注意那些外挂编码器在推理时也是要加载到显存/内存里的，所以实际 **运行时总加载约 538M 参数**...但还是很小啊 (；´∀｀)

## 环境搭建过程

### 踩坑记录

第一个坑就来了。装好 PyTorch 2.6 之后发现根本跑不起来，报了个奇怪的 CUDA 错误。

查了一下发现 RTX 5060 是 Blackwell 架构（sm_120），而 PyTorch 2.6 的 CUDA 内核只支持到 sm_90（Ada Lovelace）。需要升级到 **PyTorch 2.7+cu128** 或更高版本才能识别。

解决方法就是把 PyTorch 卸载重装：

```
pip uninstall torch torchvision torchaudio -y
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu128
```

### 模型资源下载

整个项目需要下载的资源大概 1.6GB 左右，包括语音编码器、视觉编码器、音频编解码器、说话人编码器，以及最重要的模型权重（227MB）。

下载完之后目录结构大概是这样的：

```
model/
├── SenseVoiceSmall/    (447MB)
├── siglip2-base-p32-256-ve/ (180MB)
├── mimi/               (183MB)
├── campplus/           (13.5MB)
├── speaker/            (内置音色)
└── vad/                (语音活动检测)

out/
├── sft_omni_768.pth    (227MB) ← 主要权重
├── sft_omni_768_moe.pth (610MB) ← MoE 版本
├── llm_768.pth         (基座模型)
└── sft_zero_768.pth    (零阶段权重)
```

## 推理体验

### 命令行模式

最简单的测试方式就是跑一段命令行推理：

```
python eval_omni.py --load_from model --weight sft_omni --mode 0
```

默认会用英文问 7 个问题，模型会输出文本回答并生成对应的语音帧。

![命令行推理的运行截图](/assets/images/posts/MiniMind-O初体验/cli_inference.png)

说实话文本回答的质量嘛...emmm 很符合 0.1B 的水平 (；一_一)

比如说它的回答会有句式重复、逻辑跳跃的问题，有时候前言不搭后语。但考虑到它的体量，能说出完整的英文句子已经不错了（笑）

### Gradio Web UI

然后试了 Gradio 版的 Web 界面：

```
python web_demo_omni.py --load_from ../ --port 8888
```

打开 `http://localhost:8888` 就能看到一个聊天界面。

![Gradio 版的实时语音对话界面](/assets/images/posts/MiniMind-O初体验/gradio_ui.png)

支持文本、上传语音文件、上传图片，模型会回复文本 + 合成语音。还内置了 12 种音色可以切换。

### 视频通话 (｡˃ ᵕ ˂｡)♪

这个是我觉得最惊艳的部分。

项目还带了一个 Flask 版的 Web UI，支持实时视频通话：

```
cd webui
python web_demo.py --load_from ../ --port 7860
```

打开 `http://localhost:7860/call`，打开摄像头，按住说话键，模型就会实时捕捉视频画面 + 语音输入，然后通过语音回答你。

![Flask 版 Web 的视频通话界面](/assets/images/posts/MiniMind-O初体验/video_call.jpg)

在轻薄本的 **AMD 880M 核显** 上居然也能跑起来，虽然帧率不算高，但确实能实时互动...一个 0.1B 的模型在核显上做视频通话，想想还挺魔幻的 (°▽°)

### 图片分析能力测试

既然是全模态，那肯定要测测它的图片理解能力。

**测试一：初音未来 × JavaScript 海报**

![初音未来 JavaScript 宣传海报分析](/assets/images/posts/MiniMind-O初体验/image_analysis_miku.png)

结果嘛...它完全没认出来这是初音未来 (´;ω;｀) 描述里面有很多重复和错误，比如反复说"a person with a hat"（那是头发啊喂！），而且对海报上的文字识别基本是乱猜的。

不过这也能理解，毕竟 0.1B 的视觉理解能力摆在那里，再加上训练数据里可能也没有二次元浓度太高的内容（笑）

**测试二：Linux 作业截图**

![Linux 终端作业截图分析](/assets/images/posts/MiniMind-O初体验/image_analysis_linux.png)

这张明显好一些，能认出是终端界面、命令行操作，大概能描述出"有人在执行 Linux 命令"，虽然具体的命令内容和输出结果描述得不太准确，但至少方向是对的。

## 在轻薄本（核显）上跑的体验

说实话我一开始没抱太大期望。核显跑深度学习模型？感觉就像是拿自行车上高速 (´･_･`)

但实际跑下来：

- **文本对话**：毫无压力，秒回
- **语音输入**：需要几秒钟处理，但可以接受
- **图片问答**：上传图片后大约 3-5 秒出结果
- **视频通话**：核显也能跑，画面实时捕获 + 语音流处理

当然，因为模型本身很小，回答的"智商"确实有限。你问它 "What is the meaning of life?" 它可能会给你一些看起来很合理但细想又很空洞的回答 ( ´∀｀)σ

但问题在于：**一个能在核显轻薄本上做实时视频通话的全模态模型，这本身就很有意义了。**

## 一些主观感受

### 优点 (｀・ω・´)b

- **硬件门槛极低**：核显都能跑，几乎等于没有门槛
- **全模态架构**：文本、语音、图像、视频一把抓，设计很完整
- **搭建简单**：跟着步骤走，基本上半小时到一小时就能跑起来
- **开源友好**：代码风格清晰，没什么黑盒封装，适合学习
- **项目完整度**：从训练到推理到 Web UI 一条龙，还有实时视频通话

### 缺点 (´-ω-`)b

- **智商感人**：0.1B 就是 0.1B，别指望它能帮你写代码或者做复杂的推理
- **英文为主**：虽然理论上支持中英文，但默认 prompt 和训练数据偏英文，中文对话效果更弱
- **语音质量一般**：合成语音能听，但距离自然流畅还有距离
- **视频帧率偏低**：核显上视频通话勉强可用，离"流畅"还有差距
- **图片理解薄弱**：对非自然图像（海报、二次元等）的理解能力有限

## 总结

MiniMind-O 这个项目让我想起一个词——"螺蛳壳里做道场"。

0.1B 的参数规模，硬是把文本、语音、图像、视频四种模态揉到了一起，还能在核显轻薄本上跑起来。虽然每个单项拿出来都不算强，但 **能在一个模型里同时做到这四件事**，本身就是技术上很有意思的尝试。

对于想了解全模态模型是怎么一回事的人来说，这是个很好的学习材料 (｡•̀ᴗ-)✧

毕竟，能亲手在自己电脑上跑一个能看、能听、能说、能聊的模型，本身就是一件挺酷的事情对吧？
